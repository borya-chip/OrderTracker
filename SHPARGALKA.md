# Шпаргалка по проекту OrderTracker

## 1) Что это за проект
Это REST API на Spring Boot для работы с заказами (`orders`).

Сейчас реализованы только GET-операции:
- получить все заказы (с опциональным фильтром по датам);
- получить заказ по `id`.

## 2) Главные понятия простыми словами (детально)

### Backend
Серверная часть приложения, которая:
- принимает HTTP-запросы;
- выполняет логику;
- работает с БД;
- возвращает ответ (чаще всего JSON).

В твоем проекте backend запущен на Spring Boot и слушает порт `8080`.

### API
Контракт между клиентом и сервером: какие запросы можно отправлять и какие ответы придут.

API включает:
- маршруты (URL);
- HTTP-методы (`GET`, `POST` и т.д.);
- формат параметров;
- формат ответа и коды статуса (`200`, `404`).

### Endpoint
Конкретная операция API: `метод + путь`.

Примеры из проекта:
- `GET /api/v1/orders` - получить список заказов;
- `GET /api/v1/orders/{id}` - получить один заказ по id.

### HTTP-метод GET
Тип запроса для чтения данных (без изменения состояния сервера).

Когда ты открываешь URL в браузере или делаешь `curl` на эти маршруты, вызывается именно GET.

### URL path vs query params
Два способа передать данные в запросе:

1. Path (часть пути):  
`/api/v1/orders/1` -> `1` это идентификатор ресурса.

2. Query params (после `?`):  
`/api/v1/orders?startDate=2026-02-15&endDate=2026-02-28` -> параметры фильтра.

### Controller
Входная точка HTTP-слоя.

Что делает контроллер:
- принимает запрос;
- читает параметры (`@PathVariable`, `@RequestParam`);
- вызывает сервис;
- формирует HTTP-ответ.

У тебя это `OrderController`.

### Service
Слой бизнес-логики.

Зачем он нужен:
- чтобы не держать логику в контроллере;
- чтобы можно было тестировать правила отдельно;
- чтобы переиспользовать логику из разных контроллеров.

У тебя `OrderServiceImpl` решает, какой запрос к БД делать в зависимости от переданных дат.

### Repository
Слой доступа к данным (DAO-уровень).

У тебя `OrderRepository extends JpaRepository<Order, Long>`, и это дает:
- готовые CRUD-методы (`findById`, `findAll` и т.д.);
- возможность писать методы по имени (`findByDateBetween`), а Spring сам генерирует SQL.

### Entity (доменная модель)
Java-класс, который Hibernate связывает с таблицей БД.

У тебя:
- `Order` - Entity;
- таблица - `orders`;
- поля класса -> колонки таблицы.

### DTO
Data Transfer Object - объект для обмена с клиентом.

Зачем нужен DTO:
- не отдавать наружу всю Entity как есть;
- управлять форматом API;
- изолировать внешний контракт от внутренних изменений модели.

У тебя в ответах используется `OrderDto`.

### Mapper
Преобразователь между Entity и DTO.

У тебя:
- из `Order` делается `OrderDto` перед отправкой клиенту;
- это держит контроллер/сервис чище и избегает ручного копирования полей.

### База данных (DB)
Место хранения данных.

Сейчас используется H2 in-memory:
- хранит данные в памяти процесса;
- удобна для разработки и демо;
- данные исчезают после остановки приложения.

### SQL
Язык запросов к БД (`select`, `insert`, `update`, `delete`).

В логах ты видишь SQL, который реально ушел в БД.

### ORM
Object-Relational Mapping - идея "объекты Java <-> таблицы БД".

Польза:
- работаешь с объектами, а не с низкоуровневым JDBC-кодом в каждом месте;
- меньше шаблонного кода;
- проще сопровождать проект.

### JPA
Java Persistence API - стандарт (интерфейсы и аннотации) для ORM в Java.

Важно: JPA - это не библиотека с реализацией SQL сама по себе, это именно стандарт.

### Hibernate
Реализация JPA, которая:
- читает аннотации Entity;
- строит SQL;
- выполняет запросы;
- маппит результаты обратно в Java-объекты.

Именно поэтому в логах ты видишь строки вида:
`Hibernate: select ... from orders ...`.

### Spring Data JPA
Надстройка Spring над JPA/Hibernate для ускорения разработки.

Дает:
- `JpaRepository` с готовыми методами;
- генерацию запросов по имени метода;
- интеграцию с транзакциями Spring.

### ResponseEntity и HTTP-статус
`ResponseEntity` позволяет явно управлять HTTP-ответом.

У тебя:
- успешные ответы -> `200 OK`;
- если заказа нет -> `404 Not Found`.

### Exception (исключение)
Механизм обработки ошибок в Java/Spring.

У тебя в сервисе при отсутствии заказа кидается `ResponseStatusException(HttpStatus.NOT_FOUND, ...)`, из-за чего клиент получает корректный `404`.

## 3) Как идет запрос по слоям
Путь одного GET-запроса:

1. Клиент (браузер/curl/Postman) отправляет HTTP-запрос.
2. `OrderController` принимает его.
3. Контроллер вызывает метод сервиса.
4. Сервис обращается к репозиторию.
5. Репозиторий через Hibernate выполняет SQL в БД.
6. Данные возвращаются обратно: `Repository -> Service -> Controller`.
7. Контроллер отдает JSON-ответ клиенту.

Короткая цепочка для защиты:
`Controller -> Service -> Repository -> Hibernate -> DB -> JSON`.

## 4) Твои текущие endpoints

### 4.1 GET с `@PathVariable`
`GET /api/v1/orders/{id}`

Пример:
`GET /api/v1/orders/1`

`id` берется из пути URL.

### 4.2 GET с `@RequestParam`
`GET /api/v1/orders?startDate=...&endDate=...`

Параметры передаются после `?` в query string.

Примеры:
- `/api/v1/orders` - все заказы;
- `/api/v1/orders?startDate=2026-02-20` - с указанной даты и позже;
- `/api/v1/orders?endDate=2026-02-20` - до указанной даты;
- `/api/v1/orders?startDate=2026-02-15&endDate=2026-02-28` - диапазон дат.

## 5) Почему `404` на `/orders/5`
Потому что записи с `id=5` нет в БД.

В сервисе используется:
- `findById(id).orElseThrow(...)`
- если записи нет -> бросается `ResponseStatusException(HttpStatus.NOT_FOUND, ...)`
- клиент получает `404 Not Found`.

Это корректное поведение.

## 6) Какая БД используется сейчас
Сейчас подключена **H2 in-memory** (в памяти), а не PostgreSQL.

Признаки в `application.properties`:
- `spring.datasource.url=jdbc:h2:mem:order_tracker...`
- `spring.h2.console.enabled=true`

Что это значит:
- БД живет только пока запущено приложение;
- остановил приложение -> данные в памяти пропали;
- запустил снова -> схема/данные поднимаются заново.

## 7) Откуда берутся 3 тестовых заказа
Из `src/main/resources/data.sql`.

При старте Spring выполняет SQL из этого файла и вставляет 3 записи.

## 8) Что делает Hibernate в логах
Когда ты видишь в консоли:
- `select ... from orders ...`

это значит backend реально сделал SQL-запрос в БД.

Для защиты это хороший аргумент:
"Запрос дошел до БД, видно SQL в логах Hibernate."

## 9) Команды для быстрой демонстрации преподавателю

### Запуск приложения
```bash
./mvnw spring-boot:run
```

### Проверка GET запросов
```bash
curl http://localhost:8080/api/v1/orders
curl http://localhost:8080/api/v1/orders/1
curl "http://localhost:8080/api/v1/orders?startDate=2026-02-15&endDate=2026-02-28"
```

### Остановка приложения
`Ctrl + C` в терминале, где запущен `spring-boot:run`.

## 10) Что делает Docker команда
`docker compose --env-file .env up -d --build`

Расшифровка:
- `compose` - читает `docker-compose.yaml`;
- `--env-file .env` - берет переменные из `.env`;
- `up` - запускает сервисы;
- `-d` - в фоне;
- `--build` - пересобирает образы перед запуском.

Зачем Docker:
- одинаковый запуск на разных машинах;
- удобно поднимать приложение + БД как набор сервисов.

## 11) Мини-словарь (очень коротко)
- **HTTP**: протокол общения клиента и сервера.
- **GET**: получить данные.
- **JSON**: формат ответа API.
- **Status 200**: запрос успешный.
- **Status 404**: ресурс не найден.
- **SQL**: язык запросов к БД.
- **ORM**: связывает объекты Java и таблицы БД.
